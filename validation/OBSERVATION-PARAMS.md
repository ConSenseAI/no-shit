This document turns FIXTURES §7's four sketches into concrete parameter tables at row grain, so the freeze pins numbers rather than inventing them. It fixes **structure now** and leaves **numerics for the freeze** — the same registration discipline as its parents (PROTOCOL §2: "the *structure* of every decision rule is fixed by this document now; only the named numeric parameters await the corpus audit"). Every window length, trial count, census depth, and keyword list below is a **proposal with a one-line justification**; none is decided. What *is* decided here is the shape of each rule: that longitudinal facts carry pre-registered windows, that absence claims cite the window and the census they were measured against (O-2), that rate facts carry a pre-registered `n` and a decision rule (O-4), and that rendered-appearance rows bind a vision-capable pool member (O-7). The freeze commit (PROTOCOL §2) replaces each proposed number with a pinned one and records it as a protocol appendix; nothing here may move after a sealed-corpus result is known.

Two derivations run underneath every table. **Fixture-side** parameters are exact: the fixture platform's virtual clock (FIXTURES §2.1) drives billing, scheduling, and messaging under one T0-relative **time script** recorded in the manifest, so a longitudinal fact is asserted at an *exact* scripted position (O-1 collapses weeks-to-annual into minutes), and a constructed rate is *known*, so `n` is chosen to detect it with a stated probability. **Live-cell** parameters are pre-registered wall-time windows and trial counts priced into the cell's cost model (PROTOCOL §10); a live window shorter than the fact's boundary yields **`unobserved` (partial scope → CONDITIONAL), never `pass`** — the `t:probe` rows (§1, class W-17) exist precisely to enforce this, and O-2 makes the census/window a recorded property of every clean label.

The corpus rows are byte-identical across that gap, so ATTAINABILITY §3's row classification is current. Where the corpus row text and ATTAINABILITY's channel tag diverge, §5 flags it rather than silently reconciling.

## 2. Surface-census definition (O-2)

A clean verdict is an **absence** verdict (O-2), and an absence is decidable only against an enumerated scope. This section proposes that scope: the set of surfaces a probe must sweep before it may claim a control, signal, or message is *absent*. A missed surface on a violating product is a **false PASS** — the 5%-gate error — so the census is the direct lever on that gate. The census is **recorded per fixture in the manifest** (FIXTURES §8 `census:`), and every absence claim is **scoped to it**; the sealed set's clean labels carry the census they were labeled against (O-2, PROTOCOL §9 execution-verified basis).

**General censused set (all criteria).** Proposed classes, swept on every audit:

1. **Settings / account / billing surfaces** — the full settings tree: account, profile, subscription/plan, billing, payment methods, privacy, data, security, notifications.
2. **Help / docs search** — the in-product help center and public docs, queried over the **per-criterion keyword list** below (the mechanism that finds a control documented-but-not-in-nav).
3. **Terms / privacy / legal** — ToS, privacy policy, data-processing and cookie policies, and any linked legal pages (where a disclosure or a masquerade often hides).
4. **Footers** — global and section footers (a frequent home for "export", "delete", "privacy", "unsubscribe" links).
5. **Primary-nav sweep to depth d** — breadth-first from the primary navigation to **proposed depth d = 3** (nav → section → page).

**Proposed sweep depth d = 3**, justified from the corpus: the deepest placement of a *clean* deciding control in the public rows is settings→billing→cancel and settings→account→delete (≈ 2–3 hops). A **buried** control (`adv-syn-buried-view`: cancel at 9 nav steps, never linked from billing) is not meant to be caught by *deepening* the nav sweep — it is caught by the **unbounded keyword search** (class 2) and then adjudicated as *burial* (nst check 3 / nli check 5), not as *absence* (nst check 1 / nli check 4). So the two mechanisms are complementary and their split is deliberate: the depth-3 nav sweep locates where clean controls live; the keyword search (not depth-limited) locates a present-but-buried control so it is failed for burial rather than false-flagged as missing. **An absence claim requires both the depth-d nav sweep and the keyword search to come up empty** — e.g. `viol-syn-no-export-anywhere` ("no export control anywhere — settings, help, or docs") is a true absence only after settings + help + docs all miss.

**Code-mode census (ndp code cells).** The census is the **submitted source tree** — templates, string/asset resources, route handlers, cart/consent/enrollment logic — swept exhaustively (grep-complete), not a nav walk. The keyword lists still steer where to look (timer/countdown/stock/review-filter/consent-TTL/cart-mutation code). An absence claim in code mode ("no urgency-signal generator in source") is scoped to the submitted tree.

**Per-criterion keyword lists (help/docs search; proposals; 10–20 terms; pinned at freeze).** Justified by where each corpus's *deciding facts* live.

## 3. Rate-fact trials (O-4)

Seven rows turn on a **reliability rate**, not a single observation ("frequently fails to arrive," "expires in minutes," "instant and never expires"). A rate claim from `n` trials is a statistical claim; `n` must be fixed in advance or the fact is soft (O-4). ATTAINABILITY §3 flags exactly seven **R** rows — 2 `nst`, 0 `ndp`, 5 `nli`. Note the split that drives the sizing: **5 are violating** (detect an unreliable mechanism) and **2 are clean** (certify a reliable one) — the two directions have opposite power requirements, so the decision rule is asymmetric.

Each R row decomposes into a **timing** sub-fact (link expiry — deterministic, one well-timed probe, handled in §1 W-9/W-10) and an **arrival/delivery rate** sub-fact (this section). Only the arrival rate needs `n` trials.

**Fixture side — construct the rate, choose `n` for ≥ 99 % detection.** In a fixture the per-trial failure rate `q` is *built*, so detection is exact: P(≥ 1 failure in `n`) = 1 − (1 − q)ⁿ ≥ 0.99. Worked example — `adv-syn-expiring-export-link` ("links frequently fail to arrive"): construct arrival-failure **q = 0.5** (a fair reading of "frequently"); require 1 − 0.5ⁿ ≥ 0.99 ⇒ 0.5ⁿ ≤ 0.01 ⇒ **n ≥ log₂100 = 6.64 ⇒ n = 7**; check 1 − 0.5⁷ = 1 − 0.0078 = **0.9922 ≥ 0.99** ✓. So the fixture harness issues **7** link requests and the probability all 7 arrive (masking the defect → a false PASS) is 0.78 % < 1 %. If a subtler construction is wanted to test probe sensitivity, **q = 0.3 ⇒ n = 13** (1 − 0.7¹³ = 0.990). The non-delivering shape (`adv-syn-elapsed-no-artifact`, built **q = 1.0**) is detected by **n = 1** with certainty, but is run at **n ≥ 3** (the corpus's "initiated three times") to separate *never-delivers* from *slow-but-working*. The freeze pins each fixture's (q, n) pair.

**Live side — proposed default n = 8, exact binomial bound reported.** Where the rate cannot be constructed, run **n = 8** and report the observed failures `k/8` with the **exact Clopper–Pearson (one-sided 95 %) bound** — the same point-estimate-plus-exact-CI discipline PROTOCOL §7 applies to the kill gates. Power of n = 8:

- **Detection (violation) direction:** P(catch) = 1 − (1 − q)⁸ = **99.6 %** at q = 0.5, **94.2 %** at q = 0.3, **83.2 %** at q = 0.2. Well-powered for a "frequent" failure.
- **Certification (clean) direction:** 0 failures in 8 gives a one-sided 95 % upper bound of 1 − 0.05^(1/8) = **31.2 %**. That is *weak* — "0/8 clean" only rules out q > 31 %.

**Sanity check and better proposal.** n = 8 is a sound floor for the **detection** direction — which is the false-PASS-protecting direction, and 5 of the 7 R rows live there — but it under-powers **clean certification**: a 31 % upper bound does not certify a "reliable" mechanism. Two remedies, proposed jointly:

1. **Split the sub-facts.** The clean rows' load-bearing claim is often *non-expiry*, which is **deterministic** — one probe at `t + long` settles it. Certify non-expiry by the timing probe (§1 W-9/W-10); reserve the rate sample for *arrival*, where it is actually needed. This alone removes most of the clean-side burden.
2. **Scale `n` where a clean reliability certification is the deciding fact** (`clean-syn-email-confirm-delete`, `probe-email-cancel-reliable`). To push the 0-failure upper bound under a usable tolerance: **n = 16 → 17.1 %**, **n = 23 → 12.2 %**, **n = 29 → 9.8 % (sub-10 %)**. Proposal: **n = 8 as the universal detection floor; n ≥ 23 where a clean reliability PASS is load-bearing** (or lean on the deterministic non-expiry probe per remedy 1). The freeze pins per-fact `n`.

## 4. Vision-capable judging (O-7; binds PROTOCOL §12.3)

Eight rows are decidable only at the **rendered / interactive** layer — computed styles, z-order interception, below-fold placement, emoji/mascot depiction — not from DOM text (O-7). Two of them (`adv-syn-css-lottery`, `adv-syn-hidden-delete-control`) are engineered specifically to **defeat DOM-level review**. This section enumerates them and restates the pool-membership constraint as a rule bindable at freeze against PROTOCOL §12.3 (pool pins).

| `example_id` · crit | tier | What it needs from rendered capture | Mode note |
|---|---|---|---|
| `adv-syn-hidden-cancel` · nst | E1 | computed color of the cancel control vs background (contrast); below-fold placement | behavioral-only criterion; rendered capture required |
| `probe-tiny-decline-link` · nst | E1 | computed size (small) + contrast (low) of the decline link | rendered capture required |
| `viol-syn-ghost-decline` · ndp | E1 | computed size (8 px), contrast (light-gray on white), below-fold + outside-frame placement | `· code` route can read the CSS rule; behavioral needs the render |
| `clean-syn-emphasized-cta-legible-alt` · ndp | E1 | rendered emphasis comparison — the **pass-side** anchor (alt control full-size, high-contrast, legible) | code reads the styles; render confirms legibility |
| `bound-syn-subordinated-decline` · ndp | E1 | rendered subordination — small text-link vs large accent button, but **legible** (the conditional middle band) | the three-band lane turns on the render; code corroborates |
| `adv-syn-css-lottery` · ndp | E2 | **z-order**: higher-z-index overlay intercepting pointer events over the hit area (interactable ≠ present) | engineered to defeat DOM audit; render/interaction decides |
| `adv-syn-emoji-shame` · ndp | E1 | **emoji/mascot depiction** — weeping-face emoji, slumped grayed mascot on the decline path | depiction requires a **vision-capable** judge; code sees the asset/emoji literal |
| `adv-syn-hidden-delete-control` · nli | E2 | **z-order**: transparent overlay intercepting pointer events across the delete control's hit area (rendered but not interactable) | engineered to defeat markup-level review; render/interaction decides |

Distribution across the Stage 0 matrix: `ui*` rows land in **every behavioral cell** — `nst`×behavioral (2), `nli`×behavioral (1), `ndp`×behavioral (5). The 5 `ndp` rows also carry a `· code` route: in **code mode** the code-visible facts decide structurally (the CSS rule for ghost-decline's styling, the z-index for css-lottery, the emoji/asset literal for emoji-shame), so a code cell needs vision only where it must fall back to rendered capture; in **behavioral mode** the rendered/interactive reality is the deciding evidence and vision is required for depiction.

**Bindable rule (proposal for PROTOCOL §12.3, from FIXTURES §7.4 / O-7):**

> Any pool assigned to a (criterion × pool × mode) cell whose sealed rows include a `ui*` row that the mode decides **from rendered appearance** MUST (a) judge those rows from **rendered captures** — screenshots plus computed-style, z-order, and pointer-interaction probes, not DOM text — and (b) include at least one **vision-capable** member to adjudicate **depiction** facts (emoji/mascot). A pool lacking a vision-capable member **excludes** the affected `ui*` rows from that cell, and the exclusion — which rows, and why — is **disclosed per cell** in the study report (PROTOCOL §11). Because rendered-appearance facts are behavioral, this binds **every behavioral cell** in the Stage 0 matrix (all three criteria carry `ui*` rows); code cells decide the code-visible `ui*` facts structurally and invoke the rule only where they fall back to rendered capture. The vision requirement is recorded with the pool pins at freeze (PROTOCOL §12.3), and a cell measured by a non-vision pool with `ui*` rows excluded reports its rates on the reduced row set, flagged.

## 5. Discrepancies and open notes

Read row-by-row against the corpus `README.md` files; where the row text and ATTAINABILITY §3's channel table diverge, flagged here rather than silently reconciled (ATTAINABILITY §8 already states its `t:X` tags are single-classifier authored judgment over composite rows).

1. **`t:X` tag = observation window, not mechanism duration.** ATTAINABILITY's time tags are generally the *observation* window (boundary + re-observation margin), while the corpus row states the *mechanism* duration. The two notable offsets: **trial rows** are tagged `t:8d` but the rows state a **7-day trial** (`viol-syn-trial-silent`, `clean-syn-trial-notified`, `clean-syn-trial-double-reminder`) — the 8th day is the margin to catch the conversion charge; and **`viol-syn-pause-resumes`** is tagged `t:35d` but the row states a **30-day pause** — the extra 5 days catch the silent resume. Benign, but it means FIXTURES §7.1's "8d-class" and "35d-class" labels are *window* names, not row-fact durations. Recorded so the freeze pins the window (8–9 d, ~35 d), not the row's 7 d / 30 d.

2. **`t:` bucket vs W flag are not perfectly aligned.** Three rows carry a sub-7-day or day-scale `t:` tag yet are **W-flagged** (ATTAINABILITY's "≥ 7-day live window"): `clean-syn-immediate-prorated` (`t:d`, W), `clean-syn-true-deadline` (`t:d`, W), `viol-booking-pressure` (`t:d`, W), and `bound-syn-recurring-flash-sale` (`t:3d`, W). The W flag appears to price the *cross-session / post-expiry re-observation* that pushes a day-scale fact past the ~week line, not the single `t:` bucket. Consistent with ATTAINABILITY §8's stated single-classifier caveat; flagged so the freeze prices these live windows on the observation they actually require (a billing-boundary span for `immediate-prorated`; a multi-day seriality watch for `recurring-flash-sale`) rather than the literal `t:` bucket.

3. **`clean-syn-regulated-retention` — the "5-year" figure is not a window.** ATTAINABILITY tags it `t:window` (W). The row's 5-year AML retention is a **disclosed carve-out read from text**, not an observation to wait out; the observed window is the ordinary deletion-honored window (W-6/W-11). Do not let the freeze price a 5-year live wait off the `t:window`/W tag.

4. **`bound-syn-stale-toasts` carries a data-age fact with no `t:` tag.** ATTAINABILITY §3.2 lists it `ui net`, E2, **B** — no time channel — yet the row turns on orders being "3–9 days old." The age is **seeded backend state** (backdated timestamps, FIXTURES §2.3), not a waited window, so the absent `t:` tag is defensible; noted because the row still belongs to the backend-data-age class (W-15) and is live-unattainable (O-3), which the bare `ui net` tag does not signal.

5. **SPEC header versions lead their changelogs by one minor version.** The three SPEC headers read 0.1.11 / 0.1.9 / 0.1.7 (cited here and by the task), but each changelog tops out at 0.1.10 / 0.1.8 / 0.1.6 — the T-13 calibration-gate migration, committed as HEAD~`465607d`. No 0.1.11 / 0.1.9 / 0.1.7 changelog entry exists, and the T-13 entries state "No check, decision, or corpus changes." A doc-hygiene note, not a substantive one: the corpus rows and their labels are unchanged, so §1–§4 are unaffected. Recorded for the freeze's version-reconciliation pass.

6. **The `ndp` `· code` routes on `ui*` rows create a mode-dependent vision requirement.** Five of the eight `ui*` rows are `ndp` dual-mode. In code mode several are decidable from source (CSS rule, z-index, emoji literal), so the §4 vision constraint is strictly a **behavioral-cell** requirement plus any code-cell rendered-capture fallback — not a blanket code-cell requirement. Flagged so the freeze does not over-bind the enclave code pool with a vision requirement its mode does not need (though a vision-capable enclave member remains desirable for the fallback path).
